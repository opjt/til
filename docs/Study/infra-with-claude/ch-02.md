## GCP 인증

GKE 환경을 구성하기 전에 gcloud CLI를 설치한다.

왜 CLI 프로그램을 사용할까?

GCP 계정을 생성하면서 들어가봤겠지만 GCP 콘솔에서 모든 작업을 진행할 수 있다.  
하지만 CLI 프로그램을 사용하는 이유는 AI-Agent가 웹사이트에 접속하여 클릭을 수행하는 것보다 명령어를 호출하여 작업을 수행하는 것이 더 효율적임.

```bash

```

## GKE 클러스터 생성

Notiflex의 어플리케이션을 올릴 쿠버네티스 클러스터를 설정한다.

- 쿠버네티스를 쓰는 이유는 대규모 컨테이너를 안정적으로 운영하기 위해서다, 따라서 초기 팀에서 쿠버네티스로 인프라를 셋업하는 것은 오버엔지니어링으로 여겨질 수
  있다.

`Spot VM`은 일반 VM보다 저렴한 대신, GCP가 필요할 때 회수 가능한 VM으로 AWS의 Spot Instance와 같은 개념이다.

- 클라우드 제공 회사들은 데이터센터에 idle 서버가 많이 생기기 때문에 이런 서버를 알뜰하게 사용하기 위해 싸게 빌려주는 것임
- 단점은 언제든지 종료될 수 있다는 점으로 종료 30초 전에 알림을 준다.
- 프로덕션 환경에서는 안정성 문제로 위험하지만, 학습 환경에서는 비용 절감 가능

클러스터 타입이 Standard로 설정되는 것을 알 수 있는데, 다른 타입은 없는지 질문한다.

> 클러스터 타입이 Standard 말고 있음?

GKE는 Autopilot 모드도 지원한다

- 오토파일럿은 AWS 의 Fargate와 대응한다, 노드를 안 보고 컨테이너만 배포한다는 서버리스 컨테이너 모델이다
- Standard는 노드를 직접 관리하지만. 오토파일럿은 GCP에서 노드를 관리하여 Pod 배포에만 신경 쓸 수 있다.

학습에는 노드를 직접 설정하고 관리하는 것이 학습에 의미가 있기 때문에 Standard 를 사용한다.

### 클러스터 상태 확인

```bash
# context 생성
export KUBECONFIG=~/.kube/configs/gke-notiflex.yaml
gcloud container clusters get-credentials notiflex-cluster \
    --zone=asia-southeast1-a \
    --project=opjt-gitaiops-project

# context 이름변경
kubectl config delete-context gke-sysnet4admin_book_gitaiops 2>/dev/null || true
kubectl config rename-context \
    gke_opjt-gitaiops-project_asia-southeast1-a_notiflex-cluster \
    gke-sysnet4admin_book_gitaiops
```

가드레일 설정에 kubectl 명령 사용시 `--context`로 명시하도록 되어있지만 `--context` 없이 직접 입력하고 싶은 경우

```bash
kubectl config use-context gke-sysnet4admin_book_gitaiops
```

kubeconfig을 설정하여 GKE 클러스터를 가리키도록 한 후 클러스터가 정상적으로 만들어졌는지 확인한다.

## Notiflex 앱 배포

가상의 서비스인 `Notiflex`을 배포하기 위해 Go API 서버 작성을 진행한다.

Go를 사용하는 이유는?

- 정적 바이너리 컴파일이 가능 -> scratch 베이스 이미지 가능
  - Python, Node.js, Java 는 런타임이 필요하기 때문에 불가
  - Dockerfile이 단순함
- 표준 라이브러리만으로 HTTP 서버 구현 가능 (코드가 쉬움, 인프라에 집중)

초기 Notiflex 서비스는 http 엔드포인트 2개만 제공하는 매우 간단한 기능만 갖고있다

- healthCheck 엔드포인트
- pod id 식별 가능한 엔드포인트

코드 작성이 완료되면 배포를 위한 `Dockerfile`을 작성한다.

### Dockerfile 작성

```Dockerfile
FROM golang:1.25-alpine AS builder
WORKDIR /app
COPY go.mod ./
COPY app/main.go ./
RUN CGO_ENABLED=0 go build -o api .

FROM scratch
COPY --from=builder /app/api /api
ENTRYPOINT ["/api"]
```

`scratch` 이미지는 완전히 빈 이미지를 의미한다.

컨테이너 이미지는 layer들의 스택이고 각 레이어는 tar로 묶인 파일시스템들이다(컨테이너 실행 시 레이어들을 하나의 루트파일 시스템처럼 보이게함)

`scratch`는 실제 이미지가 아니라 특수 예약어로 아무 레이어도 pull하지 않고 빈레이어로 시작하게 된다.

장점은 빈 상태이다 보니 취약점이나 공격표면 노출이 적다는 점, 단점은 기본적인 쉘도 없어서 디버깅이 어렵다

### 빌드 및 배포

빌드를 내 로컬에서 진행하지 않고 `Cloud build`를 통해 진행한다.

만약 내가 집접 하면 빌드하고, 이미지를 레지스트리에 올리고 해야하지만

`gcloud builds submit` 명령어로 통해 빌드와 이미지 업로드까지 한꺼번에 가능하다

이후 매니페스트(deployment, namespace, service)를 작성하면 배포준비는 완료된다

- [서비스 종류별 차이](https://github.com/maeilham/backend-ops/blob/main/content/0003-kubernetes-service-types.md)

```bash
kubectl apply -f k8s/smb/namespace.yaml
kubectl apply -f k8s/smb/
```

네임스페이스를 먼저 생성하고 Deployment/Service를 적용한다.

#### 배포 확인

```bash
# pod 확인
$ k get pods -n notiflex
NAME                           READY   STATUS    RESTARTS   AGE
notiflex-api-c755fd698-87kzb   1/1     Running   0          65s
notiflex-api-c755fd698-qk7sg   1/1     Running   0          65s
```

Pod 정상기동 확인 후 port-forward로 붙은 뒤 api를 테스트한다

```bash
kubectl --context gke-sysnet4admin_book_gitaiops port-forward svc/notiflex-api -n notiflex 8080:80
# 다른 터미널에서
$ curl -s http://localhost:8080/id | jq .
{
  "id": 9,
  "pod": "notiflex-api-c755fd698-qk7sg"
}
```

api를 여러번 입력해도 계속 똑같은 pod에게 요청이 간다.

port-forward를 하는 시점에 Service가 Pod 하나를 선택해서 그 파드에 직접 TCP 터널을 연결하기 때문에 이후 모든 요청은 그 터널을 통해 같은 Pod로 간다.

```bash
$ kubectl --context gke-sysnet4admin_book_gitaiops run -it --rm test \
    --image=curlimages/curl --restart=Never -- \
    sh -c 'for i in $(seq 1 6); do curl -s http://notiflex-api.notiflex/id; echo; done'
{"id":18,"pod":"notiflex-api-c755fd698-qk7sg"}

{"id":19,"pod":"notiflex-api-c755fd698-qk7sg"}

{"id":3,"pod":"notiflex-api-c755fd698-87kzb"}

{"id":4,"pod":"notiflex-api-c755fd698-87kzb"}

{"id":20,"pod":"notiflex-api-c755fd698-qk7sg"}

{"id":5,"pod":"notiflex-api-c755fd698-87kzb"}

```

이렇게 실제 서비스를 사용하도록 임시 파드를 생성하여 테스트가 가능하다.
